### Notes

Built the full todo app with the stretch goal (checkbox + cross-out via `classList.toggle`). Three files: HTML, JS, CSS. The whole thing is an exercise in the data-as-source-of-truth pattern.

`todo.js`:

```javascript
// Source of truth: tasks live here as { text, done } objects
const tasks = [];

const input = document.getElementById("task-input");
const addBtn = document.getElementById("add-btn");
const clearBtn = document.getElementById("clear-btn");
const list = document.getElementById("task-list");
const emptyMessage = document.getElementById("empty-message");

// Render the list from the tasks array. Clear-and-redraw.
function renderTasks() {
  list.innerHTML = "";

  for (const task of tasks) {
    const li = document.createElement("li");

    const checkbox = document.createElement("input");
    checkbox.type = "checkbox";
    checkbox.checked = task.done;
    checkbox.addEventListener("change", () => {
      task.done = checkbox.checked;
      renderTasks();
    });

    const span = document.createElement("span");
    span.textContent = task.text;
    if (task.done) {
      span.classList.add("done");
    }

    li.appendChild(checkbox);
    li.appendChild(span);
    list.appendChild(li);
  }
}

// Add a task
addBtn.addEventListener("click", () => {
  const newTask = input.value.trim();
  if (newTask === "") {
    emptyMessage.classList.remove("hidden");
    return;
  }
  emptyMessage.classList.add("hidden");
  tasks.push({ text: newTask, done: false });
  input.value = "";
  renderTasks();
});

// Clear everything
clearBtn.addEventListener("click", () => {
  tasks.length = 0;
  renderTasks();
});

// Allow Enter key to add tasks
input.addEventListener("keydown", (event) => {
  if (event.key === "Enter") {
    addBtn.click();
  }
});
```

### Why this architecture matters

The pattern looks small but it's load-bearing:

1. **One source of truth.** All app state lives in the `tasks` array. The DOM is never the source of truth, it's a reflection.
2. **Mutations go to the data first.** Add button pushes to `tasks`, then calls `renderTasks()`. Clear button sets `tasks.length = 0`, then calls `renderTasks()`.
3. **Rendering is a pure function of state.** `renderTasks()` wipes the list and rebuilds it from whatever's in `tasks` right now. It never reads from the DOM.

That's the React mental model in vanilla JS. Once this clicks, `useState` and the re-render cycle are just a more ergonomic version of the same pattern: change state, re-render the view.

### About `const tasks`

`const tasks = []` says "the variable `tasks` will always point to this same array." It does NOT say "this array can never change." `.push()`, `.pop()`, `tasks.length = 0` all mutate the contents of the array, and `const` is fine with that. What `const` blocks is `tasks = somethingElse` (reassigning the variable). The binding is locked, the contents are not.

### What I learned

Data-driven rendering means you can stop thinking about the DOM as something to manipulate piece by piece. You change the state, you re-run the renderer, and the DOM gets rebuilt to match. Mutations get easier (push to an array, splice, etc.) instead of harder (find the right element, update it, hope it's in sync). This is the entire reason React exists, and you can do it in vanilla JS with one clear-and-redraw function.
