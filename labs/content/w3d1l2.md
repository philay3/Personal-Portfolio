### Notes

Exercise 2 was getting the project from "scope document" to "live URL." Three things: create the repo, wire it to a deploy target, and get the initial commit pushed so changes start flowing automatically.

### Repo setup

Created `philay3/Personal-Portfolio` on GitHub, cloned locally, and committed three empty starter files:

```
Personal-Portfolio/
├── index.html
├── style.css
└── script.js
```

### Deployment

Connected the repo to Vercel through the Vercel dashboard, then pointed my custom domain `philanthony.dev` at the project. Every push to `main` redeploys automatically. That part is the biggest workflow upgrade compared to manual GitHub Pages publishing. I write, commit, push, and 30 seconds later the live site reflects the change.

### Initial commit

```bash
git add .
git commit -m "Initial project setup: HTML, CSS, JS files"
git push
```

Verified on GitHub that the files were there, then opened philanthony.dev to confirm Vercel had picked them up.

### Later structural change

After the initial setup I added `/labs` as a subfolder inside this same repo for Next Chapter lab writeups (the page you're reading lives there). Originally I'd started a separate `philay3.github.io` repo for that, but consolidating into one portfolio repo is cleaner. One deployment, one domain, one place to push.

### What I learned

Two parts of "setup" felt over-engineered until I did them once:

- Custom domain at the deploy target. Took five minutes in Vercel's dashboard. Now the URL on my resume isn't a random `*.vercel.app` subdomain.
- Auto-deploy from main. The push-to-deploy loop is short enough that I trust it. I don't double-check the live site after every commit. I just commit, and if something broke, the next commit fixes it.

Both are the kind of tooling that disappears once it's working, which is the sign that the setup was worth doing.
