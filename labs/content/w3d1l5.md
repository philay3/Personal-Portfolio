### Notes

The final exercise of Day 1 was adding one interactive feature to the portfolio. The lab listed six options: hamburger menu, scroll-to button, content filter, form validation, theme toggle, typing animation. I built a scroll spy on the navigation. As you scroll the page, the nav link for the section currently in view highlights.

I wrote about the build process for this one on Substack: [Rebuilding a Scroll Spy with the Intersection Observer API](https://philanthony3.substack.com/p/rebuilding-a-scroll-spy-with-the).

### Why this feature

Three reasons scroll spy beat the others:

- The site is single-page with anchor navigation, so nav state matters more here than on a multi-page site.
- It introduced me to the Intersection Observer API, which I hadn't used before, and which is the modern replacement for scroll-position math.
- It's a "first impression" feature. Visitors see it the moment they start scrolling, even before they read any project copy.

The lab options were a menu, not a checklist. A hamburger menu would have made sense if I had a denser nav, but I have four links and they all fit on one row.

### How it works (in plain English)

1. Grab every section element on the page (`#biography`, `#projects`, `#articles`, `#gallery`).
2. Create one `IntersectionObserver` that watches all of them.
3. The observer fires whenever a section crosses into or out of the viewport.
4. When a section enters the viewport, find the nav link whose `href` matches that section's id, add a `.is-active` class to it, and remove the class from all the other nav links.
5. CSS does the rest. `.site-nav__links a.is-active` gets the accent color.

### The core code

```javascript
const sections = document.querySelectorAll("main section");
const navLinks = document.querySelectorAll(".site-nav__links a");

const observer = new IntersectionObserver(
  (entries) => {
    for (const entry of entries) {
      if (entry.isIntersecting) {
        const id = entry.target.id;
        for (const link of navLinks) {
          link.classList.toggle("is-active", link.getAttribute("href") === `#${id}`);
        }
      }
    }
  },
  {
    // Fire when the section is roughly centered in the viewport,
    // not the instant its top edge crosses in.
    rootMargin: "-40% 0px -40% 0px",
    threshold: 0,
  }
);

for (const section of sections) {
  observer.observe(section);
}
```

### What tripped me up

- The first version highlighted multiple links at once. Two sections were "intersecting" at the same time when the viewport was tall enough to show both. The `rootMargin: "-40% 0px -40% 0px"` fixed this. It shrinks the observer's "active zone" to a horizontal strip in the middle of the viewport, so only one section is considered "in view" at a time.
- CSS specificity caught me. My first `.is-active` rule wasn't winning against the base link styling. I had to bump specificity by scoping it: `.site-nav__links a.is-active { color: ... }`. Lesson re-learned: a class on its own often loses to a more specific selector earlier in the file.
- Anchor jumps weren't smooth. Added `html { scroll-behavior: smooth; }` in CSS. One line, big difference. Didn't need JavaScript for it.

### Key takeaways from the build

- Scope ruthlessly before writing any code. Every "nice to have" I cut on Day 1 was a day I didn't burn later. Gallery is still empty and that is fine, because it was always nice-to-have.
- AI generates structure. Voice is mine. AI gave me clean semantic HTML and reasonable CSS defaults. What turns it into a portfolio instead of a template is real names, real projects, real copy. The lorem ipsum has to die in the first edit pass.
- One real project beats three fake ones. The Digital Footprinter card is doing the heavy lifting on the projects section because it's actually live, actually mine, and actually has a story.
- Pick interactivity that solves a real problem on your site. Scroll spy made sense for a long single-page layout. A hamburger menu would have made sense if I had a denser nav. The lab options were a menu, not a checklist.
- Vercel plus custom domain is the workflow upgrade. Push to main, site updates. No FTP, no manual deploys, no separate build step. Once it's wired up it disappears, which is what good tooling does.
