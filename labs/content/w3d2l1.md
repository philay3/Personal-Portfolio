### The plan-as-comment trick

I dropped today's plan as a code comment at the top of `script.js`:

```javascript
// TODAY'S PLAN:
// 1. Finalize CSS for hero, projects, articles, gallery sections
// 2. Make the layout responsive at the 700px breakpoint
// 3. Build the scroll spy with Intersection Observer
// 4. Smooth scroll on anchor clicks
// 5. Test on mobile (Chrome DevTools device toolbar + real phone)
// 6. Commit after each milestone
```

Each item got crossed out (in the comment) as I finished it. Dumb, low-tech, effective. Two reasons it works for me:

- The plan lives next to the code I'm editing. No second app, no separate doc.
- Crossing items off feels good and keeps me from drifting into nice-to-haves while must-haves were still incomplete.

The discipline is in step 6. Every working feature gets its own commit so I can roll back surgically if something later breaks.

### What I learned

The morning of a build day is the cheapest hour to spend on structure. Five minutes writing the plan paid for itself before lunch. Twice this morning I caught myself opening a tab I didn't need, and the comment block at the top of the file pulled me back.
