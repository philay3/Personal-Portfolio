### Notes

The peer activity at the end of the day was a verbal mini-demo. Pair up, two-minute walkthrough, use code as the artifact. I used the scroll spy. The exercise was as much about explaining the decisions as it was about showing the result.

### Demo structure

What I showed:

- The live site scrolling through all four sections, nav links lighting up for each one.
- The source code in VS Code, walking through the Intersection Observer setup.

The four points I made walking through the code:

- `document.querySelectorAll("main section")`: grabs every `<section>` element in `<main>`. Plural API, returns a NodeList.
- The observer callback: fires whenever any observed section crosses the threshold I set.
- `link.classList.toggle("is-active", booleanExpression)`: the two-argument trick. Pass a boolean and `toggle` adds the class when true, removes when false. Cleaner than `if/else`.
- `rootMargin: "-40% 0px -40% 0px"`: explained as the bug fix. Without it, two sections would be considered "in view" simultaneously when the viewport was tall enough. With it, only sections in the middle 20% of the viewport count.

### Partner feedback

"I understood ___": that the scroll spy is built on the browser firing events when sections enter and leave the viewport, and that I use those events to update the nav.

"I have a question about ___": what happens when the user lands on the page? Does anything highlight before they scroll? Good catch. On page load, the section already in view triggers an immediate callback, so the right nav link lights up before any scrolling happens. Intersection Observer fires once on attach for any section currently intersecting.

That kind of question is exactly the prep I needed. The Day 4 presentation isn't going to be "here's my code." It's going to be "here's a decision I made and why."

### End-of-day reflection

What worked today:

- Commit after every feature. I rolled back twice when changes broke the layout, and each rollback was painless because the bad commit was isolated.
- The morning plan as a comment. Crossing items off a comment block at the top of `script.js` is dumb and effective. Kept me from drifting into nice-to-haves while must-haves were still incomplete.
- Debug first, re-prompt second. I hit the scroll spy multi-highlight bug, opened DevTools, watched the observer fire, realized two sections were intersecting simultaneously, fixed `rootMargin` myself. If I'd thrown it at AI immediately I'd have gotten a generic answer instead of understanding the root cause.

What I'd do differently:

- Start the responsive pass earlier. I left mobile to the end of the CSS checkpoint and ended up making layout decisions twice. Better to consider both viewports while writing the original CSS.
- Trust the scope. I almost lost an hour debating whether scroll spy was must-have or nice-to-have. The scope said nice-to-have. I should have just trusted the document I wrote yesterday and moved on faster, then come back to it once must-haves were done, which is what I eventually did.
