### Peer demo: one design decision, walked through

I ran my partner through the live site at philanthony.dev, then walked through one design decision: choosing scroll spy as the interactive feature over a hamburger menu.

Why scroll spy, not hamburger: the site is single-page with anchor navigation. A hamburger menu solves dense-nav-on-mobile, which I don't have (four links isn't dense). Scroll spy solves "where am I right now on this long page," which is a real question when the page is one continuous scroll. Picking the feature that solves an actual problem on my site beats picking the feature that's most common in tutorials.

Partner feedback:

- "The clearest part was ___": the reasoning for picking scroll spy over the other options. Framing it as "what problem does this solve on my specific site" landed.
- "I'd love to hear more about ___": what the next feature I'd add would be, and why. Good question. My honest answer: an actual gallery, populated with one or two more projects so the projects section isn't carrying the whole portfolio alone.

That's also the feedback I'm taking into the Day 4 presentation. Be explicit about what's done, but also be honest about what's next. Hiring managers and reviewers prefer "here's what I'd add next and why" over "this is finished."

### Change tracking: Day 3 additions

Carrying over the running log from Day 2 and appending today's polish-pass additions:

- Per-section padding rules repeated across multiple class blocks. Consolidated into CSS custom properties (`--space-xs` through `--space-xl`) in `:root`. DRY. One place to change spacing across the whole site.
- AI suggested adding a complex animated mobile menu. Did nothing. Four nav links don't need a hamburger. Adding one would be solving a problem I don't have.
- Default font sizes that didn't establish hierarchy. Set explicit sizes for H1, H2, H3 with intentional ratios. Type hierarchy is a design decision, not a default.
- AI suggested adding loading skeletons for images. Skipped. Two images on the page, both small. Skeletons would add complexity for no real user benefit on a fast site.

### End of Week 3 reflection

What clicked this week:

- Scope is destiny. The scope I wrote on Day 1 set the boundaries for everything else. When I felt the pull to add features mid-week, the scope said no, and that was the right answer.
- Commit after every feature actually pays off. I rolled back twice this week. Both rollbacks were painless because the bad commit was isolated.
- AI as collaborator, not AI as author. The site looks like mine, sounds like mine, makes the decisions I'd make, because I treated every AI output as a draft to revise. The change-tracking log is the proof.

What I'd do differently on the next project:

- Start with a real-device test, not at the end. I caught one bug only on real-phone testing (the nav wrap). If I'd been testing on a phone throughout, I'd have caught it earlier and not had to refactor at the end.
- Write the README earlier. I haven't written one for this repo yet. Adding it to my Day 4 cleanup list.
