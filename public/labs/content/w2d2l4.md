### Notes

Asked AI to build a small page that doubles a number on button click, with comments on every line. Then read the JavaScript line by line and tracked decisions in an accept/reject/modify column. Most lines were accept. The one modify: switched result.innerHTML to result.textContent because the output was plain text and didn't need HTML parsing (also safer). Also confirmed the Number() cast on input.value was correct, since input values are always strings by default and skipping the cast would have caused string concatenation bugs (5 + 5 = '55' instead of 10).

### Code

```
Decision log:

Line: const input = document.getElementById("num-input");
  What: Grabs the input element by ID for JS to interact with.
  Decision: Accept
  Why: Standard, most efficient way to access a single element in vanilla JS.

Line: event.preventDefault();
  What: Intercepts the default form submission.
  Decision: Accept
  Why: Essential. Without it the browser refreshes and the user never sees the result.

Line: const doubled = Number(input.value) * 2;
  What: Casts the input string to a Number and multiplies by 2.
  Decision: Accept
  Why: input.value is always a string. Without the cast, "5" * 2 might work due to coercion, but "5" + 5 would give "55". Explicit cast is safer.

Line: result.innerHTML = ...
  What: Injects the result string into the paragraph.
  Decision: Modify
  Why: Switched to .textContent. Output is plain text, so textContent is safer (no XSS risk) and slightly more performant (no HTML parsing).
```

### What I learned

Reading AI-generated code line by line with intent to accept/reject/modify is the workflow. Most lines are fine. The few that aren't tend to be either security concerns (innerHTML for plain text) or correctness concerns (missing type casts on user input).
