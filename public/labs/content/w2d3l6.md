### Notes

Practice writing a good bug description and a bad bug description, then comparing how AI responds to each.

**The good description**

Used Bug 4 from Exercise 4 (the string-vs-number temperature bug). Wrote up the bug with expected behavior, actual behavior, error message status, and the relevant code:

> I'm working on a small HTML page that prompts the user for a temperature in Fahrenheit and displays a message describing the weather.
>
> What I expected: If I enter `9`, the page should show "It's cold! Bundle up!" because 9 is below 60.
>
> What actually happens: Entering `9` shows "It's hot outside!" The message logic seems to think 9 is greater than 85.
>
> Error message: None in the console.

```javascript
const temp = prompt("Enter the temperature in Fahrenheit:");
if (temp > 85) {
  message = "It's hot outside!";
} else if (temp > 60) {
  message = "Nice weather!";
} else {
  message = "It's cold! Bundle up!";
}
```

AI immediately identified that `prompt()` returns a string and that `"9" > "85"` does lexicographic comparison (character by character, where `"9"` comes after `"8"`). Suggested wrapping `prompt()` in `Number()` or `parseInt()`. Explanation matched what I worked out on my own.

**The bad description**

Pasted the same code with the message "my code doesn't work". AI's response was much vaguer. It listed everything that could be wrong instead of zeroing in. It pointed out the missing `let` on `message` (a real but unrelated issue) and suggested restructuring the whole `if/else`. Without knowing the symptom (entering `9` returns "hot"), it couldn't find the actual bug.

### What I learned

The more specific the symptom (input X gives output Y, expected Z), the faster AI converges on the real problem. "Doesn't work" makes AI guess. The expected-vs-actual contrast is the single most useful thing to include in any bug report, to AI or to a human.
