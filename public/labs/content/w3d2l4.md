### Notes

Checkpoint 3 was the mid-afternoon milestone: one interactive feature working. I built a scroll spy on the nav with the Intersection Observer API. As you scroll the page, the nav link for whichever section is currently in view picks up an `.is-active` class and lights up in the accent color.

The deeper walkthrough of why-this-feature lives in w3d1l5 and on Substack ([Rebuilding a Scroll Spy with the Intersection Observer API](https://philanthony3.substack.com/p/rebuilding-a-scroll-spy-with-the)). This entry is the build-day record: when it landed, what tripped me up live, and the commit.

### The core implementation

```javascript
const sections = document.querySelectorAll("main section");
const navLinks = document.querySelectorAll(".site-nav__links a");

const observer = new IntersectionObserver(
  (entries) => {
    for (const entry of entries) {
      if (entry.isIntersecting) {
        const id = entry.target.id;
        for (const link of navLinks) {
          link.classList.toggle("is-active", link.getAttribute("href") === `#${id}`);
        }
      }
    }
  },
  {
    rootMargin: "-40% 0px -40% 0px",
    threshold: 0,
  }
);

for (const section of sections) {
  observer.observe(section);
}
```

### Two bugs encountered live, one CSS bonus

Hit and fixed in real time:

- Multiple links highlighting at once. Two sections were "intersecting" simultaneously when the viewport was tall enough to show both. Fixed by tightening `rootMargin` to `-40% 0px -40% 0px`, which shrinks the observer's "active zone" to a horizontal band in the middle of the viewport.
- CSS specificity not winning. My first `.is-active` rule was getting overridden by the base link styling. Scoped it as `.site-nav__links a.is-active` and it took.

One CSS bonus from the same checkpoint, no JavaScript needed:

```css
html { scroll-behavior: smooth; }
```

One line and the anchor jumps stop being abrupt.

### What I learned

The biggest learn here wasn't the API itself. It was the order of operations. AI's first instinct for "highlight nav on scroll" was a scroll-event listener doing `getBoundingClientRect` math. That works, but it's the old pattern. Asking "is there a more modern way to do this?" before accepting the first answer is becoming a habit.

The other learn was debugging-first. I hit the multi-highlight bug, opened DevTools, watched the observer fire in the console, and figured out the root cause myself. If I'd thrown it back at AI right away I'd have gotten a generic answer. The fix was specific to my viewport heights and section sizes, and the only way to find it was to look.
