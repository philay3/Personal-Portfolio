**Why `e.preventDefault()`:** a form's native submit triggers a full page reload (the pre-JavaScript behavior). In a React SPA that wipes all state. `preventDefault()` stops the browser default so the handler controls what happens instead.

    category={product.category}
  />

**Why React needs the `key` prop**

When the array changes, React must match the previously rendered rows to the new data: was something removed, reordered, added? `key` gives each row a stable identity so React diffs correctly and touches only what changed. With no key, or with the array **index** as the key, deleting item 2 shifts every later row's identity. React reads it as "row 3's content changed" instead of "row 2 was deleted," producing wrong-row checkbox states and lost focus in editable lists. Rule: stable id (`key={product.id}`), never the index, for any list that can change.

### Exercise 5: To-do app (single component version)

Built with all three extensions: All/Active/Completed filter, Clear Completed (shown conditionally via `todos.some(t => t.completed)`), and styling hooks.

**Trace: the three required walkthroughs**

**1. Type "Buy groceries" and click Add**
- Each keystroke: controlled-input loop. `onChange` reads `e.target.value`, `setInputValue` updates state, re-render shows the character
- Click Add -> form `onSubmit` fires `addTodo(e)`
- `e.preventDefault()` blocks the native page reload
- Guard: empty/whitespace input returns early. **No setState, no re-render**
- New object built: `{ id: Date.now(), text: "Buy groceries", completed: false }`
- `setTodos([...todos, newTodo])`. New array via spread
- `setInputValue("")` clears the box; one re-render reflects both updates; count shows 1 remaining

**2. Check the checkbox**
- `onChange` fires `toggleTodo(id)`
- **Double immutability:** `.map()` builds a new *array*, and the matching item is replaced with a new *object*: `{ ...todo, completed: !todo.completed }`. Untouched todos pass through unchanged; the changed one is rebuilt, never edited in place
- Re-render: strikethrough applies via the ternary `textDecoration`, `remainingCount` (derived, computed each render) drops to 0, Clear Completed button appears (`hasCompleted` flips true)

**3. Click Delete**
- `deleteTodo(id)` -> `todos.filter(t => t.id !== id)`. New array without the item
- List empties; `todos.length === 0` conditional renders "No tasks yet. Add one above!"

### Exercise 6: Component decomposition

Final structure:

```
TodoApp                (owns todos, filter, isAlphabetical - all shared state)
├── TodoForm           (owns its OWN inputValue; sends final text up via onAdd)
├── TodoFilters        (activeFilter + onFilterChange props)
├── TodoSorted         (isAlphabetical + onToggle props - Codex CLI feature)
├── TodoList           (todos, onToggle, onDelete -> maps to TodoItem)
│   └── TodoItem       (todo, onToggle, onDelete - one row)
└── TodoStats          (remainingCount)
```

```jsx
// TodoForm.jsx - local state for the input; only the committed text leaves
import { useState } from "react";

export default function TodoForm({ onAdd }) {
  const [inputValue, setInputValue] = useState("");

  function handleSubmit(e) {
    e.preventDefault();
    if (inputValue.trim() === "") return;
    onAdd(inputValue.trim());
    setInputValue("");
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="text"
        value={inputValue}
        onChange={(e) => setInputValue(e.target.value)}
        placeholder="Add a new task..."
      />
      <button type="submit">Add</button>
    </form>
  );
}
```

```jsx
// TodoItem.jsx
export default function TodoItem({ todo, onToggle, onDelete }) {
  return (
    <li>
      <input
        type="checkbox"
        checked={todo.completed}
        onChange={() => onToggle(todo.id)}
      />
      <span style={{ textDecoration: todo.completed ? "line-through" : "none" }}>
        {todo.text}
      </span>
      <button onClick={() => onDelete(todo.id)}>Delete</button>
    </li>
  );
}
```

```jsx
// TodoList.jsx
import TodoItem from "./TodoItem";

export default function TodoList({ todos, onToggle, onDelete }) {
  return (
    <ul>
      {todos.map((todo) => (
        <TodoItem key={todo.id} todo={todo} onToggle={onToggle} onDelete={onDelete} />
      ))}
    </ul>
  );
}
```

```jsx
// TodoStats.jsx
export default function TodoStats({ remainingCount }) {
  return (
    <span>
      {remainingCount} item{remainingCount !== 1 ? "s" : ""} remaining
    </span>
  );
}
```

(TodoFilters and TodoSorted as built: array-driven buttons with conditional active class; full TodoApp wires everything via props.)

**Where state lives, and why (gate-ready answer)**

State lives at the **lowest component that needs it**. Only TodoForm cares what's currently being typed, so `inputValue` is local to it. But `todos` is needed by TodoList, TodoStats, and the filters (siblings), so it must live in their shared parent, TodoApp. That's "lifting state up." Data flows down as props; actions flow up as callback props (`onAdd`, `onToggle`, `onDelete`).

**Sort note:** `.sort()` mutates the array it's called on. `[...filteredTodos].sort(...)` copies first, so state is never mutated. `localeCompare` handles alphabetical comparison correctly.
